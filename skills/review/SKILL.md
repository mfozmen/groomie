# Groomie — review a breakdown

Run the groomie self-review over an **existing** breakdown, standalone. This is the same
verification pass the groom and revise flows run automatically after emitting their files —
exposed as its own command for outputs produced earlier (or by an older groomie version).

The checklist itself is canonical and lives in the main skill:
**`../groomie/references/review-checklist.md`** (resolve against this skill's base path —
sibling directory). The rule text behind every item lives in
`../groomie/references/breakdown-guide.md`. Read BOTH before judging anything; do not
review from memory.

## Flow

1. **Resolve the breakdown.** From the arguments take the issue key. Locate the pair
   `<KEY>/<KEY>-groomed.md` **and** `<KEY>/<KEY>-groomed.json` (per-issue folder), falling
   back to the legacy flat `<KEY>-groomed.{md,json}` in the working directory. Both files
   must exist together at one location — if not, tell the user to run `/groomie <KEY>`
   first and stop. Never review (or regenerate) from memory of a previous session.
2. **Load the contract.** Read `review-checklist.md` and the guide sections it points to.
3. **Check.** Read the `.md` (and `.json`) top to bottom once, answering every checklist
   item. Collect all violations before fixing anything.
4. **Fix — one bounded pass** (the checklist's "Fixing" section governs): minimal edits in
   the `.md` and `.json` together, keys stable, `jira` ledger untouched; regenerate the
   `.html` by the template concat from the main skill's *Output markdown* step, and re-run
   check-graph — resolving both against **this skill's base path** (the harness gives it to
   you; substitute it for `$SKILL` below, and note the sibling hop: assets at
   `$SKILL/../groomie/assets/`, checker at
   `node "$SKILL/../groomie/scripts/check-graph.mjs" <json> <md>`). Never use bare relative
   paths — they resolve against the shell's working directory, not this skill. Both are
   best-effort like everywhere else: if `node` can't run, eyeball the graph invariants; if
   the assets can't be found, skip the `.html` and say so. Anything
   needing new research or a judgment call the ticket doesn't answer is **reported, not
   fixed** — never iterate a second pass, never invent content.
5. **Report.** One line if clean (`Self-review: clean — N stories, M tasks checked`);
   otherwise the fixed items and the reported-but-not-fixed items, by key. If the JSON's
   `jira.pushed` ledger is non-empty, add the drift warning: the Jira issues now lag these
   fixes until the user runs `/groomie:push <KEY>` again. When files were rewritten, **end
   the message with the main skill's closing block** (step 5 of a fresh groom): the file
   paths last, plus the nothing-written-to-Jira note.

## Boundaries

- **Read-only against Jira** — this skill never calls the Atlassian MCP and never pushes;
  fixes land only in the local files.
- **The verification items are shape checks here.** You have no ticket and no research sources,
  so the checklist's "nothing unverified stated as fact" / "contradictions read as both-sided
  open questions" items are judged on what the files say — never re-verify a claim, never open
  new research. (In the groom and revise flows, which do have both, they are backed by the
  verification the main skill's step 3 already ran.)
- Never hand-author the `.html`; it is only regenerated by the template concat (skip it
  and say so if the assets can't be found).
- One fix pass, then stop. A breakdown that needs restructuring beyond mechanical fixes
  (stories that shouldn't exist, missing research) is a `/groomie:revise` or a fresh
  `/groomie <KEY>` — say which, don't do it here.
- If invoked with no issue key, ask for one before doing anything else.
